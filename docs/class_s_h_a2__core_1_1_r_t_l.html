Detailed Description

Architecture of the hash core.

The core is composed by two parallel and distinct pipelines, the Expander pipeline and the Compressor pipeline. Each stage of the Compressor pipeline is paired with a ROM containing the \(K\) constant values needed for that stage. The pipeline depth can be configured by the generic parameter PIPELINE_STAGES, which disables pipelining if set to 1.

Overall data path. Operative part in blue, Control part in red

Pipeline clocking

The pipeline registers are employed also as round registers, hence are clocked by the external base clock. The two working modes are selected by a multiplexer placed in front of each pipeline register, which switches between the feedback of the output from the current stage and the output from the previous stage. This multiplexer is driven by the carry output of the rounds counter, which consequently works as a stage clock and is hence called end_major_cycle.

Pipeline loading

Since the signal end_major_cycle drives the input to the pipeline registers, and in particular to the first stage, the architecture is sensitive to its primary input M_blk only when end_major_cycle is asserted. For this reason, the signal end_major_cycle must be propagated to the primary output ready.

Upon reset, the signal end_major_cycle is cleared, hence the pipeline is not loaded during the first major cycle of operations. To avoid losing this major cycle, an additional signal first_major_cycle is introduced and or-ed with end_major_cycle in the first stage. It is not necessary in the subsequent stages, which are in indifference condition during the first major cycle, and during all subsequent major cycles the signal end_major_cycle is properly asserted and drives the data exchange as designed. Moreover, since the padded data block can be removed from the primary input, it must be stored in an input register only for the very first major cycle. A multiplexer, driven by the signal first_major_cycle, commutes between the primary input and the buffered version, which is excluded during the subsequent major cycles in order not to lose an additional clock cycle.

It is worth noting that, during each major cycles, the output is captured by the pipeline register of the stage during all but the last step, whereas the output of the last step is captured by the pipeline register of the following stage. The last step is the one with end_major_cycle asserted, hence the one with the counting value equals to 0. This means also that the first computation executed within the stage is the one with the counting value equals to 1, since the computation with the previous value. is discarded by the multiplexer. This allows for compensating the one-cycle delay introduced by the ROM. The signal init ensures this behaviour also for the very first major cycle.

Tempification issue with reordering

The signals end_major_cycle and init are both generated by the Control Unit and are actually the same signal if FIX_TIME is set to false. On the other end, if the value of the \(K\) constants is consumed in the compressor pipeline stage ahead of the pipeline registers, the Control Unit instantiated by defining FIX_TIME to true adds a latency state for the compressor pipeline only, hence differentiating these two signals.

Moreover, when FIX_TIME is set to true, the Expander pipeline works with a major cycle delayed of one clock cycle w.r.t. the major cycle of the Compressor pipeline, in order to take into account the additional latency of the Compressor pipeline.

Pipeline register

Apart from the hash variables, the register stores the flag valid which is asserted if the register contains valid data. This flag is used to produce the output signal completed, since the content of the output register is valid, representing the output of a previously provided input, if and only if the valid flag is asserted in the output register.

The valid flag is initialised in the first stage by the start primary input. To ensure the proper setting of the flag also for the very first major cycle, this primary input is buffered with the same logic of the input message.

The final stage

Differently from the other stages, the final sum stage requires only one clock cycle. This final sum can optionally been performed in a separate stage. When this is the case, the output register of this stage, which is also the output register for the whole circuit, is delayed of one clock cycle by means of a flip-flop.

If there is at least one adder before the pipeline register in the Compressor pipeline stage, however, the separate stage is not profitable and can hence be disabled.

Supporting a variable number of stages

In order to be capable of supporting a configurable number of pipeline stages, a counter for pipeline stages is added. This counter is modulo PIPELINE_STAGES and is used by the Control Unit to determine when the pipeline has been fully flushed. The clock signal of this counter is the major cycle clock, i.e. end_major_cycle.

Another issue in supporting a configurable number of stages is the K_ROM, the content of which must be defined for each stage, for every supported configuration. PIPELINE_STAGES determines the number of ROMs in which to partition the SHA-2 ROM, whereas UNROLLING_FACTOR determines the width of the ROMs.

Pipelining and the Expander

In the implementation of SHA-2 with pipelining, it must be taken into account that the word required at each step depends up on word values of up to 16 cycles ahead.

One option is to precalculate all words, keeping into account that the computation is entirely combinatorial. However, in this case this computation must be done within the first major cycle, in order not to increment the latency of the circuit.

The other way is to split the expander into stages. The parallel output of the shift register of the expander becomes the parallel input for the shift register of the expander of the next stage. The combinatorial operations on the feedback line becomes the stage of the expander pipeline.
